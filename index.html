01 \u2014 Origem
De onde veio o projeto

Como uma planilha de controle manual se tornou um sistema web completo rodando inteiramente no browser.

O Problema Original

O controle operacional da recep\u00e7\u00e3o da unidade Smart Fit era feito de forma manual: planilhas, anota\u00e7\u00f5es em papel, WhatsApp para pend\u00eancias. Dados de alunos novos, vendas de addons, NPS e escala de trabalho n\u00e3o tinham uma fonte \u00fanica da verdade.

Cada recepcionista registrava informa\u00e7\u00f5es de forma diferente. Relat\u00f3rios mensais levavam horas para compilar. Pend\u00eancias se perdiam entre conversas.

A Decis\u00e3o

A solu\u00e7\u00e3o foi criar um sistema digital pr\u00f3prio. A escolha de come\u00e7ar como um \u00fanico arquivo HTML foi deliberada: sem servidor, sem deploy, sem depend\u00eancias externas. Qualquer computador com um browser poderia abrir e usar imediatamente.

Isso eliminou a barreira de entrada e permitiu itera\u00e7\u00f5es r\u00e1pidas. O arquivo poderia ser passado por pen-drive, WhatsApp ou e-mail.

Escopo Inicial \u2014 O que precisava existir

Operacional

Registro de alunos novos com atendente respons\u00e1vel, vendas de addons vinculadas aos atendimentos, controle de pend\u00eancias com status e respons\u00e1veis.

Gest\u00e3o

Dashboard executivo com m\u00e9tricas do m\u00eas, NPS com ranking de recepcionistas, escala de trabalho mensal com regras trabalhistas.

Infraestrutura

Segrega\u00e7\u00e3o por per\u00edodo mensal, exporta\u00e7\u00e3o e importa\u00e7\u00e3o de backups JSON, sistema de fechamento mensal que avan\u00e7a o per\u00edodo automaticamente.

02 \u2014 Conceito
O que \u00e9 o sistema

WPM Gest\u00e3o Interna \u2014 um sistema operacional completo de gest\u00e3o de recep\u00e7\u00e3o rodando 100% no browser, sem backend.

Defini\u00e7\u00e3o T\u00e9cnica

O sistema \u00e9 uma Single Page Application (SPA) single-file \u2014 um \u00fanico arquivo .html com todo o HTML, CSS e JavaScript embutidos (~11.500 linhas na v34). O estado \u00e9 armazenado em IndexedDB como backend prim\u00e1rio e localStorage como espelho de fallback, segregado por per\u00edodo no formato YYYY-MM. N\u00e3o h\u00e1 servidor, n\u00e3o h\u00e1 banco externo, n\u00e3o h\u00e1 login.

A arquitetura \u00e9 dividida em camadas expl\u00edcitas: config \u2192 persist\u00eancia \u2192 schema \u2192 dom\u00ednio/seletores \u2192 renderiza\u00e7\u00e3o \u2192 UI/eventos \u2192 diagn\u00f3sticos. Todo o contrato de atualiza\u00e7\u00e3o de estado \u00e9 expl\u00edcito: A\u00e7\u00e3o \u2192 Muta\u00e7\u00e3o \u2192 saveData() \u2192 requestRender().

Per\u00edodos Mensais

O sistema trata cada m\u00eas como uma unidade isolada. Dados de 2026-01 nunca vazam para 2026-02. O sistema suporta navega\u00e7\u00e3o completa entre per\u00edodos hist\u00f3ricos com dados preservados.

Campos deriv\u00e1veis \u2014 como dia da semana a partir de uma data \u2014 nunca s\u00e3o salvos no storage. S\u00e3o calculados em tempo de execu\u00e7\u00e3o para economizar quota de armazenamento.
